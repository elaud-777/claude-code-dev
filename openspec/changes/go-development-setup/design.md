## Context

이 저장소는 현재 비어 있으며 Go 관련 설정이 전혀 없다. 앞으로 만들 구체적인 서비스/도구는 미정이므로, 이번 변경은 어떤 후속 작업에서도 재사용 가능한 표준 Go 개발 기반을 마련하는 데 집중한다.

## Goals / Non-Goals

**Goals:**
- `go.mod` 기반의 표준 모듈 구조 확립
- 포맷팅(`gofmt`)과 린트(`golangci-lint`) 자동 검사 체계 마련
- `go test`를 통한 테스트 실행 규칙과 최소 예시 테스트 제공
- 로컬 및 CI에서 동일하게 사용할 수 있는 빌드/린트/테스트 명령 정의
- CI(예: GitHub Actions)에서 빌드·린트·테스트 자동 실행

**Non-Goals:**
- 특정 애플리케이션 기능(API, CLI 명령 등) 구현
- 배포/릴리스 파이프라인 구성
- 데이터베이스, 외부 서비스 연동

## Decisions

- **모듈 구조**: `cmd/<app>/main.go` + `internal/` 패키지 레이아웃 채택. 표준 Go 프로젝트 레이아웃 관례를 따라 이후 어떤 애플리케이션이 추가되더라도 확장 가능하도록 함.
- **린트 도구**: `golangci-lint` 채택 (대안: `go vet`만 사용). `golangci-lint`가 다수의 린터를 통합 제공하고 커뮤니티 표준 도구이므로 선택.
- **작업 자동화**: `Makefile` 사용 (대안: 셸 스크립트, `Taskfile`). Go 생태계에서 가장 널리 쓰이는 방식이며 별도 의존성 설치가 필요 없음.
- **CI 플랫폼**: GitHub Actions 가정 (저장소가 GitHub에 있다는 전제). 다른 CI를 쓸 경우 워크플로우 파일만 교체하면 되므로 변경 범위에 미치는 영향은 작음.

## Risks / Trade-offs

- [실제 애플리케이션 요구사항 없이 구조를 먼저 정함] → 이후 실제 기능 추가 시 레이아웃이 맞지 않으면 재조정이 필요할 수 있음. 표준 레이아웃을 따르므로 재조정 비용은 낮음.
- [CI 플랫폼 가정이 틀릴 경우] → 워크플로우 파일 위치/문법만 바뀌므로 다른 아티팩트에는 영향 없음.
